# SpaceOS 팀원 가이드 — 상권 데이터 취합·분석 (개발 경험 0, 바이브코딩 초심자용)

> **대상** — 개발 업무를 한 번도 안 해 본 팀원. 코드를 직접 짜지 않는다.
> **한 줄 요약** — VS Code에서 Claude에게 **한 줄 명령**을 주면 상권 1곳을 분석해 코드에 넣어준다. 너는 **검토·승인·업로드**만 한다.
> **협업** — 각자 자기 브랜치에서 작업 → **매주(또는 격주) 리드가 취합**한다.

---

## 0. 큰 그림 (먼저 2분만 읽기)

```
[너] 상권 1곳 배정받음
   → VS Code 열기 → 최신 내려받기 → 내 브랜치 만들기
   → "/district-add 상권명 자치구"  한 줄 실행
   → Claude가 데이터 취합·분석·코드작성 (너는 승인)
   → 커밋 → 깃허브에 올리기(push) → PR 생성
   → 다음 상권 반복
[리드] 매주 PR 모아서 취합 + 수집 배치 + 재학습 + 배포
```

### 세 가지만 기억하면 된다
1. **혼자 코딩 안 한다.** 명령 한 줄 주면 Claude가 다 한다. 너는 "응/좋아"로 승인.
2. **막히면 붙여넣어라.** 빨간 오류가 나오면 그 글자를 복사해 Claude에 붙여넣고 **"이 오류 고쳐줘"**. 2번 시도해도 안 되면 **리드에게 공유**하고 다음으로 넘어간다. (혼자 붙들지 말 것 — 멈추는 게 최악)
3. **내 것만 건드린다.** 배정받은 상권만. `main` 브랜치에서 직접 작업 금지, 남의 상권 파일 금지.

### 절대 금지 3가지 ⛔
- ❌ `main` 브랜치에서 바로 작업/저장 (항상 내 브랜치에서)
- ❌ 남에게 배정된 상권 건드리기 (배정표 확인)
- ❌ 오류를 혼자 30분 붙들기 (2번 시도 → 리드)

## 1. 최초 1회 셋업 (30~40분, **딱 한 번만**)

> 이 장은 **처음 하루에 한 번만** 한다. 다음부터는 2장(작업 루프)만 반복.

### 1-1. 프로그램 3개 설치
아래를 순서대로 설치한다. 전부 "다음 → 다음 → 설치" 기본값으로 진행하면 된다.

| # | 프로그램 | 어디서 | 용도 |
|---|---------|--------|------|
| 1 | **VS Code** | code.visualstudio.com | 작업하는 편집기 |
| 2 | **Git** | git-scm.com/download/win | 버전관리(내려받기·올리기). 설치만 하면 됨 |
| 3 | **GitHub 계정** | github.com 가입 | 리드에게 계정 아이디 전달 → 저장소 초대 받기 |

### 1-2. VS Code에 Claude 확장 설치 + 로그인
1. VS Code 왼쪽 세로 아이콘 중 **네모 4개(확장, Extensions)** 클릭
2. 검색창에 **`Claude`** 입력 → **Claude Code** 설치(Install)
3. 설치 후 안내에 따라 **로그인**(리드가 알려준 계정으로)

### 1-3. 저장소 내려받기(복제, clone) — 한 번만
1. VS Code 상단 메뉴 → **Terminal → New Terminal** (아래에 검은 입력창이 열림)
2. 아래를 **복사해서 붙여넣고 Enter**. (원하는 위치에 `spaceos` 폴더가 생긴다)
   ```powershell
   cd $HOME\Documents ; git clone https://github.com/seoghyeonbag36-max/spaceos.git
   ```
   > "Authenticate"/로그인 창이 뜨면 GitHub 계정으로 로그인한다. (리드에게 저장소 **초대(collaborator)**를 미리 받아둘 것)
3. VS Code 상단 → **File → Open Folder** → 방금 생긴 **`spaceos`** 폴더 선택 → 열기

### 1-4. 잘 됐는지 확인
- VS Code 왼쪽 파일 목록에 `CLAUDE.md`, `apps`, `data` 폴더가 보이면 성공.
- Claude 채팅창에 `/` 를 입력했을 때 목록에 **`/district-add`** 가 보이면 준비 완료.

> 💡 **수집 환경(파이썬·API키)은 없어도 된다.** 아래 "초심자 모드"로 진행하면 Claude가 공개 정보로 분석하고, 무거운 실측 수집·재학습은 **리드가 중앙에서** 대신 돌린다.

## 2. 매번 하는 작업 루프 (상권 1곳 = 아래 A~H)

> 상권 하나를 끝낼 때마다 A부터 H까지. 각 단계의 **파란 상자를 Claude 채팅창에 그대로 붙여넣으면** 된다.
> (git 명령을 외울 필요 없다. Claude가 대신 해준다.)

### A. 최신 내용 받기 (남들 작업 반영)
```
main 브랜치의 최신 내용을 깃허브에서 받아줘 (git pull). 지금 내 위치도 main인지 확인하고.
```

### B. 내 브랜치 만들기
> 브랜치 이름 규칙: `district/이름-상권` (예: `district/jimin-yeonhui`)
```
district/{내이름}-{상권} 이라는 새 브랜치를 만들고 그 브랜치로 옮겨줘.
```

### C. 상권 분석 실행 ★핵심 (명령 한 줄)
> `연희동`·`서대문구`를 **내 배정 상권**으로 바꾼다. 뒤 문장은 초심자 모드 안전장치.
```
/district-add 연희동 서대문구

참고: 나는 개발 초심자이고 파이썬/API키 실행환경이 없어. STEP3 실데이터 수집이 환경 없이
안 되면 건너뛰고, 공개적으로 알려진 정보(대표 업종·분위기·주요 골목)를 근거로 seed를 작성하되
모든 추정 수치에 "# TODO: 실측 교체" 주석을 남겨줘. STEP6 검증(pytest/build)도 리드가 할 테니 생략해도 돼.
막히면 15분 이상 붙들지 말고 TODO로 남기고 다음으로 진행해.
```

### D. 승인하며 진행
- Claude가 단계마다 "이렇게 할까요?" 물으면 내용을 훑고 **"응 진행해"** / **"좋아"** 로 승인.
- 이상하면 한국어로 그냥 말한다. 예: *"공실유닛 5개 중 3층 임대료가 너무 높은 것 같아, 근거 다시 확인해줘."*
- **빨간 오류**가 보이면 → 그 글자 복사 → 붙여넣고 **"이 오류 고쳐줘"**.

### E. 결과 눈으로 확인 (선택)
```
방금 추가한 상권이 apps/backend/app/data/seoul_pages.py 에 잘 들어갔는지, 감성구역 6개·공실유닛 5개·행사 3개가
다 있는지 요약해서 보여줘.
```

### F. 커밋 (작업 저장)
```
지금까지 변경한 내용을 커밋해줘. 커밋 메시지는 "feat: {상권명} 상권 추가" 로.
```

### G. 깃허브에 올리기 (push)
```
지금 내 브랜치를 깃허브에 올려줘(push).
```

### H. PR(취합 요청) 만들기 — 리드에게 보내는 상자
1. 브라우저에서 **github.com/seoghyeonbag36-max/spaceos** 접속
2. 노란 띠의 초록 버튼 **"Compare & pull request"** 클릭
3. 제목 자동입력 확인 → **"Create pull request"** 클릭 → 끝. (리드가 취합함)

> PR 버튼이 안 보이면 Claude에 이렇게: `방금 올린 브랜치로 PR 만드는 링크 알려줘.`

### → 다음 상권
A로 돌아가 반복. (다음 상권은 **A에서 다시 최신 받기**부터 — 중요)

## 3. 자주 나는 상황 대처법

| 상황 | 이렇게 한다 |
|------|------------|
| 빨간 글씨(오류)가 나옴 | 글자 복사 → Claude에 붙여넣고 "이 오류 고쳐줘". **2번 시도 후에도 안 되면 리드**. |
| Claude가 파이썬/키가 없어서 못 한다고 함 | 정상. C단계 초심자 모드 문장대로 "공개정보로 작성 + TODO"로 진행하라고 하면 됨. |
| "이미 main에서 작업했어요" | 당황 금지. Claude에 "지금 변경사항을 새 브랜치로 옮겨줘"라고 하면 됨. |
| push가 거부됨(rejected) | "main 최신 받아서 다시 올려줘"라고 Claude에. 그래도 빨간색이면 리드. |
| 어디까지 했는지 모르겠음 | Claude에 "지금 나 어느 브랜치이고 뭐가 커밋됐는지 알려줘". |

> **원칙: 헷갈리면 결정하지 말고 물어본다.** Claude에 한국어로, 리드에게 메신저로. 잘못 눌러 데이터가 날아가는 일은 브랜치 방식에선 거의 없다.

## 4. 상권 배정 (중복 방지) — 시작 전 리드가 채움

각자 **자기 줄의 상권만** 작업한다. 서로 다른 상권이면 취합 시 충돌이 거의 없다.

| 팀원 | 배정 상권 (10곳) | 진행 브랜치 |
|------|------------------|------------|
| (예) 지민 | 연희·망원·합정·서교·성산·상암·공덕·아현·대흥·염리 | `district/jimin-*` |
| … | … | … |

> ⚠️ **같은 상권을 두 명이 하지 않는다.** 배정표에 없는 상권을 하고 싶으면 리드에게 먼저 말해 표에 넣는다.

## 5. 【리드 전용】 매주(격주) 취합 — 팀원은 안 읽어도 됨

팀원 PR이 쌓이면 리드가 **한 자리에서** 취합·검증·배포한다.

### 5-0. 통합 원칙 — "데이터는 옮기지 않고, 재생성한다" ★먼저 이해

팀원의 수집 데이터(`data/bronze`·`data/gold`)는 **깃허브에 안 올라간다**(`.gitignore` 제외). 그런데도 통합이 되는 이유:
분석의 **판단·결과는 코드(seed)에 들어가 git으로 합쳐지고**, 원본 수집물은 **공개 API에서 결정론적으로 다시 받을 수 있는 파생물**이라 옮길 필요가 없기 때문이다(연희동을 누가 수집하든 같은 데이터).

| 종류 | 파일 | git 공유? | 통합 방법 |
|------|------|:---:|-----------|
| 분석 결과(사이트에 보임) | `apps/backend/app/data/seoul_pages.py` | ✅ | **PR 머지로 자동 통합** |
| 상권 매핑(TRDAR 코드) | `data/config/platform_districts.py` | ✅ | **PR 머지로 자동 통합** |
| 원본 수집물 | `data/bronze/*` | ❌ | **옮기지 않음 — 리드가 재수집** |
| 가공·예측 산출물 | `data/gold/*` | ❌ | **리드가 배치로 재생성** |

**→ 그래서 주간 통합은 두 갈래다.**
- **방법 A (기본):** 팀원은 코드(seed+매핑)만 PR로 넘기고, **데이터는 리드가 중앙에서 재수집**한다. 서울시 전체(수백 상권)로 확장돼도 이 방식이 정답. (아래 5-1 → 5-2(A) → 5-3)
- **방법 B (예외):** 팀원이 **재현 불가능한 작업**(현장 사진·로드뷰 수기 라벨·수동 보정)을 했을 때만, 그 원본을 **공유 드라이브(Google Drive/S3)** 로 올려 리드가 내려받아 쓴다. (아래 5-2(B))

### 5-1. PR 취합 (주 1회)
1. github.com/seoghyeonbag36-max/spaceos → **Pull requests** 탭 → 각 PR 내용 확인.
2. 저장소 루트에서 `claude` 실행 후:
   ```
   열려 있는 PR 목록을 확인하고, 상권 seed 추가 PR들을 하나씩 main에 머지해줘.
   seoul_pages.py / platform_districts.py 에서 충돌이 나면, 양쪽 상권 항목을 둘 다 살리는 방향으로 해결해줘
   (서로 다른 상권이라 삭제 없이 병합 가능). 머지 후 pytest 통과를 확인해.
   ```
   > 충돌은 대부분 "리스트 끝에 서로 다른 상권을 추가"라 **양쪽 다 남기면** 해결된다. Claude가 처리.

### 5-2. 실데이터 수집 + 검증
> 팀원이 TODO로 남긴 프록시 수치를, 리드가 키를 가진 환경에서 실측으로 채운다. 아래 A·B 중 상황에 맞게.

**(A) 중앙 재수집 — 기본 경로** (팀원은 데이터를 안 보냄)
```
data/config/platform_districts.py 에 이번에 추가된 상권들의 TRDAR 매핑이 다 있는지 확인하고,
data/collectors (seoul_trdar·kakao_local·naver_blog·rone_rent) 로 신규 상권들을 배치 수집(Bronze)해줘.
그다음 cd apps/backend && pytest, cd apps/frontend && npm run build 로 검증해줘.
PoC 함정 5항(STORES_RADIUS_M=600·PIP폴백·층별분모·업종 도메인 정합·EPSG:2097 보정)도 신규 상권에 적용.
```

**(B) 공유 드라이브로 받아 쓰기 — 재현 불가능한 원본이 있을 때만**
> 폴더 규약: 공유 드라이브 `SpaceOS-Data/bronze/{상권id}/` (팀원이 자기 수집물 업로드) → 리드가 로컬 `data/bronze/`로 동기화.
```
공유 드라이브에서 내려받은 원본이 이미 data/bronze/ 에 있다. 재수집 없이 그 데이터로 바로 진행해줘:
python -m data.pipelines.refresh_platform --skip-collect
(수집 단계를 건너뛰고 Gold 재빌드 + 학습만). 이후 검증은 (A)와 동일.
```

### 5-3. 예측 재학습 + 배포 (배치, 여기서만)
```
python -m data.pipelines.refresh_platform --skip-collect 로 Gold 재빌드 + LSTM 재학습을 돌려
data/gold/platform_vacancy_forecast.json 에 신규 상권 forecast를 채워줘.
검증 통과하면 커밋하고 main에 push 해줘 (Vercel 자동 배포).
배포 후 https://spaceos-sandy.vercel.app 에 신규 상권이 보이는지 확인.
```
> **주의:** 재학습은 리드가 **여기서만** 돌린다(팀원은 안 함). 데이터(`data/bronze`·`data/gold`)는 깃허브에 안 올라가므로, 리드가 **중앙에서 재수집**해 학습하는 것이 정상 경로다.

### 5-4. 충돌을 더 줄이고 싶으면 (선택, 리드 판단)
상권 seed를 지금은 한 파일(`seoul_pages.py`)에 모두 넣는다. 팀원이 많아 충돌이 잦으면,
상권별로 파일을 쪼개는 구조(예: `app/data/districts/{id}.py`)로 리팩터링하면 **두 명이 같은 파일을 건드릴 일이 사라진다.**
필요 시 Claude에 "seed를 상권별 파일로 분리하고 로더에서 자동 병합하게 리팩터링해줘"로 요청.

## 부록. 복붙용 프롬프트 카드 (한 상권 처리 전체)

```
① main 브랜치의 최신 내용을 깃허브에서 받아줘 (git pull).
② district/{내이름}-{상권} 새 브랜치를 만들고 옮겨줘.
③ /district-add {상권명} {자치구}
```
