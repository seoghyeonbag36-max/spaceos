## 0. 다섯 가지 원칙 (모든 프롬프트에 공통)

| # | 원칙 | 이번 PoC에서의 증거 |
|---|---|---|
| 1 | **1프롬프트 = 1바이브 사이클** — 목표 한 문장 + 산출물 정의. 여러 목표를 한 프롬프트에 넣지 않는다 | "building_vacancy.py 구현에 착수할 것" 한 문장으로 수집기→스모크→720동 완주 |
| 2 | **실측이 스펙이다** — API 문서 대신 프로브/실호출 로그를 근거로 지시한다 | D1 로그의 39개 필드가 곧 콜렉터 스키마가 됨. 문서상 필드명과 실제가 달랐음 |
| 3 | **실패 로그를 원문 그대로 붙인다** — 해석하지 말고 복사해 넣는다 | "사이트가 접속 안 된다" → LOCALDATA 폐쇄 발견. maxFeatures 예외 원문 → 상한 1000 확인 |
| 4 | **스키마의 주인을 지정한다** — 데이터를 소비하는 쪽(프론트/ML)의 타입이 단일 출처 | 백엔드·Gold를 프론트 `VacStatus` 4종(full/partial/high/empty)에 맞춤 |
| 5 | **합격 기준을 숫자로 넣는다** — "잘 되게" 대신 "±5%p, 70%+" | PoC exit 기준이 있어서 검증 스크립트까지 자동으로 만들어짐 |

**금지 2가지**: ① API 키를 프롬프트/문서에 붙이지 않는다(오직 `.env`) ② Bronze/Gold 대용량 데이터를 커밋시키지 않는다(gitignore 확인 지시).

## 1. 단계별 프롬프트 템플릿

### Phase 0 — 결정·설계 (회의 → 문서)
회의 메모나 아이디어를 **결정 근거가 남는 설계 문서**로 굳힌다.
```
[회의 메모/생각 붙여넣기]
이 내용을 docs/____.md 설계 문서로 정리할 것.
반드시 포함: ① 우리가 답해야 하는 핵심 질문(Q1·Q2 형식) ② 확정 결론 요약
③ 데이터 소스와 조인 키 ④ 알려진 한계·편향 ⑤ 검증(성공) 기준 숫자.
```
> 실전 예: 「Page 방향성」 회의 → poc-building-vacancy.md. 이 문서의 §2 공식·§3 검증 기준이 이후 모든 프롬프트의 근거가 됐다.

### (A) 인증키·외부 의존 단계
```
[기능]에 필요한 외부 API/키를 전부 조사해서 docs에 정리할 것.
발급 사이트·정확한 데이터셋 명칭(검색어 그대로)·.env 변수명·비용·승인 소요를 표로.
기존 키로 공용 가능한 것과 신규 발급을 구분할 것.
```
키를 채운 뒤에는 반드시: `두 .env에 입력된 키들이 문제가 없는지 실호출로 확인할 것.`
> 실전 교훈: 실호출 검증에서 "키가 .env.example(git 추적)에 잘못 들어간 것", "구 API 서비스 종료" 2건을 발견했다. **검증 프롬프트를 생략하면 이런 건 며칠 뒤에 터진다.**

### Phase 1 — 수집 (Bronze 콜렉터)
```
data/collectors/____.py 콜렉터를 구현할 것.
- 소스: [엔드포인트] (아래 실호출 로그의 필드 기준)
- 저장: Bronze 규칙(bronze/{거점}/{날짜}/) 준수, 원본 무가공
- 페이징: totalCount까지. 상한/쿼터: [숫자] — 초과 시 분할/부분저장
- 키 없으면 건너뛰고 로그만 남길 것(파이프라인 안전)
- 먼저 기존 콜렉터(collectors/common.py) 관례를 확인하고 맞출 것
[실호출/프로브 로그 붙여넣기]
```
> 실전 교훈: "관례를 확인하고 맞출 것" 한 줄이 저장 경로·인코딩 보정·load_env 재사용을 전부 해결한다. 스모크 테스트(`LIMIT_BUILDINGS=8` 같은 환경변수)를 요구하면 쿼터를 아끼며 로직을 먼저 검증할 수 있다.

### Phase 2 — 가공 (Silver/Gold 결합)
```
data/pipelines/____.py 를 구현할 것.
- 조인: [A파일의 키] == [B파일의 키] (형식 명시: 예. PNU 19자리 동형)
- 미매칭 처리 규칙을 명시적으로: [예: 상업용도 미점포 → 공실의심, 주거 → 제외]
- 출력 스키마는 [소비자: 프론트 타입/백엔드 서비스]가 아는 값만
- 실행 후 분포·매칭률·집계치를 출력해 공식 통계와 비교 가능하게
```
> 실전 예: "미매칭 폴리곤 중 상업용도 = 공실의심" 규칙에서 **empty 229동**이 나왔다. 미매칭 처리 규칙이야말로 이 단계의 본체다.

### Phase 3 — 백엔드 API (더미 → 실데이터)
```
apps/backend/app/services/____.py 의 더미를 Gold 파일 로드로 교체할 것.
- 응답 스키마는 기존 더미와 동일하게 유지(프론트 무수정 원칙)
- Gold 파일이 없으면 기존 더미로 폴백(다른 개발환경 보호)
- 교체 후 uvicorn을 직접 띄워 curl로 응답 건수·status 분포를 검증할 것
```

### Phase 4 — 프론트 Page
```
/page [목표 한 문장]
- 데이터가 UI를 결정: 실데이터를 먼저 조회해 분포를 보고 표현을 제안할 것
- 색·간격은 디자인 토큰만(하드코딩 금지). 네이버 그린은 연동 맥락만
- 완료 기준: npm run build 통과 + 스크린샷
```

### Phase 5 — ML (Platform)
```
/ml-train [목표]
- 입력 데이터 계약을 먼저 명시: [Gold 테이블, 키, 시계열 축]
- 데이터가 부족하면 학습 강행하지 말고 "무엇이 몇 분기 더 필요한지" 보고할 것
- 지표: [정확도 70%+ 등] 미달 시 원인 분석 포함
```

### Phase 6 — Posting·Program
Phase 1~3 패턴 재사용 + LLM 프롬프트 체인은 **Humanistic Authority(균형·공생·공감) 톤 명시**.

### STEP 7 — 검증·PoC exit
```
[기능]의 검증을 준비할 것.
- 판정 기준: [숫자. 예: 부동산원 ±5%p, 로드뷰 라벨 정확도 70%+]
- 사람이 해야 하는 라벨링은 "클릭만 하면 되는" 워크시트로 만들 것(링크 포함)
- 채점 스크립트를 같이 만들어 판정이 재현 가능하게
```

## 2. 진행상황별 프롬프트 패턴

### ① 착수 — 이미 합의된 다음 단계를 시작할 때
```
____ 구현에 착수할 것.
```
짧을수록 좋다. 단, **직전 대화에서 근거(설계 문서·로그)가 합의된 상태**여야 한다. 맥락이 없으면 Phase 템플릿으로.

### ② 막혔을 때 — 오류·접속불가·예상 밖 동작
```
[증상 한 문장 + 로그/스크린샷 원문]
문제를 파악할 것.
```
**"고쳐라"가 아니라 "파악하라"**로 시작한다. 원인이 외부(서비스 폐쇄·API 종료·정책 변경)일 수 있어서, 진단 없이 수정을 지시하면 엉뚱한 곳을 고친다.
> 실전 예: "localdata.go.kr 접속 자체가 안 된다. 문제를 파악할 것" → 사이트 폐쇄+이관 발견 → 문서·코드 일괄 반영.

### ③ 순서를 정할 때
```
남은 작업 [목록]을 순서대로 해야 하는가, 중요도 순으로 해야 하는가?
```
의존성 사슬(출력→입력)이면 순서가 강제되고, 독립 작업만 중요도/병렬 선택지가 있다. 답을 받으면 ①로 착수.

### ④ 마일스톤 저장
```
현재까지 작업을 저장할 것. (커밋 메시지에 무엇을·왜가 남게)
그리고 [체크리스트/진행 문서]를 현재 상태로 갱신할 것.
```
커밋 단위 = 논리적 변화 1개. 데이터 파일이 아닌 코드·문서만.

### ⑤ 역할 분기 확인 — 내가 뭘 해야 하는지
```
이 다음에 어떻게 해야 하는가? 사용자가 할 일과 개발이 할 일을 구분해서.
```
로그인 필요한 다운로드·현장 라벨링·콘솔 설정은 사용자 몫, 코드·검증은 Claude 몫으로 갈라진다.

### ⑥ 외부 정보가 낡았을 가능성이 있을 때
```
[문서/절차]가 지금도 유효한지 실제 접속/호출로 확인하고, 낡았으면 문서까지 고칠 것.
```
> 이번 PoC에서 3건이 낡아 있었다: LOCALDATA 폐쇄, 건축물대장 v2 종료, 국가공간정보포털→V-World 이관. **2개월 전 문서도 의심하라.**

## 3. 다음 사이클 — PoC 정확도 개선 백로그 (복붙용 프롬프트)

현재: 추정 56.1% vs 부동산원 41.6% (gap 14.5%p, α=0.742). 로드뷰 라벨 30동 대기 중.
정확도를 올리는 프롬프트, 효과 큰 순:

1. **층별개요 정밀화** (floor_approx 559동이 최대 오차원)
   ```
   building_vacancy.py의 일반건물 capacity를 "층수×2호" 근사에서
   건축HUB 층별개요(getBrFlrOulnInfo)의 상업용도 층만 카운트하는 방식으로 정밀화할 것.
   쿼터 여유를 계산해 초과 시 floor_approx 폴백 유지. 개선 전후 gap 비교 출력.
   ```
2. **가두/집합 분리 집계** (poc §5 특이점 — 앵커가 2개인 이유)
   ```
   page_building_master를 1층 도로변(가두)과 집합상가로 분리 집계해
   부동산원 가두 41.6% / 집합 9.99% 각각과 비교하는 이중 α보정을 구현할 것.
   도로변 판별은 폴리곤-도로 인접으로, 어려우면 층수·용도 기반 근사부터.
   ```
3. **좌표 PIP 매칭 폴백** (bdMgtSn 누락 5건 + 지번 불일치 회수)
   ```
   bdMgtSn/지번 미매칭 점포를 V-World 폴리곤에 point-in-polygon으로 귀속시키는
   Silver 매칭 폴백을 추가할 것. match_method 필드로 품질 추적.
   ```
4. **폐업 반영 지연 보정** (상가정보 분기 갱신 편향)
   ```
   카카오 로컬 API로 상가정보 활성 점포의 현존 여부를 크로스체크하고,
   data.go.kr로 이관된 지방행정 인허가(폐업일자)를 수집해 분자를 보정할 것.
   (localdata.py TODO — 이관 후 신규 스펙 확인부터)
   ```
5. **로드뷰 라벨 확대** — 30동 결과가 70% 근처면 표본을 60동으로 늘려 신뢰구간 확보.

## 4. 프롬프트가 실패하는 3가지 이유 (회고)

1. **근거 없이 지시** — "공실 데이터 수집해줘"는 실패한다. 어떤 소스·어떤 키·어떤 공식인지가 설계 문서에 있어야 한다. → Phase 0을 건너뛰지 마라.
2. **검증 생략** — "만들어줘"로 끝나는 프롬프트. "실행해서 분포를 출력하고 공식 통계와 비교할 것"까지가 한 사이클이다.
3. **낡은 전제** — 문서·예제 코드가 가리키는 외부 세계는 변한다. 막히면 ②(문제 파악) 패턴으로 전제부터 의심하라.
